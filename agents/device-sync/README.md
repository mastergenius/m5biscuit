# Biscuit Device Sync Host

Mac-side sync substrate for Biscuit devices. It is intentionally broader than Study Twin:
StudyPacks, review logs, notes, crash summaries, and future local-first payloads should share the
same pairing and transport model.

The server is a FastAPI app with the same firmware-facing API as the earlier stdlib prototype.

## Role

The host daemon is the target for one-button device-initiated sync:

1. M5Paper joins saved WiFi.
2. M5Paper discovers or uses a saved paired host endpoint.
3. M5Paper authenticates with a pairing secret.
4. M5Paper pushes dirty local payloads and receives acknowledgements.

The existing File Transfer QR/token flow remains the fallback path for manual recovery and file
uploads.

## Local Files

```text
agents/device-sync/
  biscuit-sync       # CLI and server launcher
  biscuit_sync_app.py
  biscuit_sync_common.py
  run                # uv-backed CLI wrapper with local cache/env
  dev-server         # hot-reload shortcut
  dev-tmux           # shared tmux session shortcut
  sync-env           # prefetch/update local uv env and cache
  smoke-test         # temporary-root FastAPI/HMAC smoke test
  config/            # local host identity and secrets, ignored by git
  inbox/             # imported payloads, ignored by git except placeholders
    notes/
    study-logs/
    device-status/
```

Secrets in `config/*.json` are local machine state and must not be committed.

## Install

```bash
agents/device-sync/sync-env
```

`run`, `sync-env`, `dev-server`, and `dev-tmux` set a local uv cache and project environment by default:

```text
agents/device-sync/.uv-cache/
agents/device-sync/.venv/
```

Both directories are ignored by git. Override `UV_CACHE_DIR` or `UV_PROJECT_ENVIRONMENT` if you
want the cache/env somewhere else.

`dev-server` uses `uv run`, so the explicit install step is optional. It exists to prefetch the
environment before running the server.

## Commands

Initialize host identity:

```bash
agents/device-sync/run init
```

Create or replace the temporary bootstrap secret:

```bash
agents/device-sync/run set-bootstrap-secret
```

Write the current bootstrap secret into the ignored firmware-local fallback:

```bash
agents/device-sync/run write-firmware-bootstrap \
  --host-url http://192.168.1.165:8787 \
  --device-id m5paper-5c013b0db9b0
```

If `--host-url` or `--device-id` are omitted, the command preserves the existing local value from:

```text
src/activities/apps/DeviceSyncDebugTemporaryPairingFallback.local.h
```

Run the FastAPI server directly:

```bash
agents/device-sync/run serve \
  --host 0.0.0.0 \
  --port 8787 \
  --advertise
```

Run with hot reload:

```bash
agents/device-sync/dev-server
```

Run or attach to a shared tmux session:

```bash
agents/device-sync/dev-tmux
tmux attach -t biscuit-sync
```

Send a one-shot debug command over USB serial after flashing firmware with serial command support:

```bash
scripts/device-command --port /dev/cu.usbserial-5B1F0123571 sync
scripts/device-command --port /dev/cu.usbserial-5B1F0123571 status
```

Useful environment overrides:

```bash
HOST=127.0.0.1 PORT=8790 agents/device-sync/dev-server
SESSION=biscuit-sync-test agents/device-sync/dev-tmux
```

Run a local end-to-end smoke test with a temporary host root:

```bash
agents/device-sync/smoke-test
```

## Temporary Firmware Bootstrap

The bootstrap flow exists to avoid SD-card editing while QR pairing is not implemented.

On first `Device Sync`, if `/biscuit/sync/host.json` is missing, unreadable, or missing required
fields such as `host_url` or `secret`, the M5Paper uses the ignored firmware-local fallback to:

1. connect to the bootstrap `host_url`;
2. authenticate with the bootstrap secret using challenge/HMAC;
3. receive a generated per-device sync secret from the server;
4. write `/biscuit/sync/host.json` on SD;
5. immediately run normal challenge/HMAC sync using the per-device secret.

The generated SD file has this shape:

```json
{
  "schema": "biscuit.device-sync-host.v0",
  "host_url": "http://192.168.1.20:8787",
  "device_id": "m5paper-5c013b0db9b0",
  "secret": "<64 hex chars generated by the host>"
}
```

`/biscuit/sync/host.json` wins when present and valid. The embedded bootstrap key is used only when
the SD pairing file is missing or cannot be used.

Security notes:

- the bootstrap secret is not sent over the network;
- the generated per-device sync secret is returned over HTTP once during bootstrap;
- payloads are not encrypted, so HTTP sync is trusted-LAN only;
- an embedded bootstrap secret can be extracted from firmware or a flash dump;
- replace this path with QR pairing and secret rotation before using sync on untrusted networks.

## API v0

Public:

- `GET /health`
- `GET /api/v0/host`
- `POST /api/v0/bootstrap/challenge` when temporary bootstrap is enabled
- `POST /api/v0/bootstrap` when temporary bootstrap is enabled

Paired device flow:

- `POST /api/v0/challenge` with `{"device_id": "..."}`
- `POST /api/v0/sync` with HMAC-authenticated payload

The v0 daemon stores incoming payloads under `inbox/` and returns acknowledgements plus a small
`summary` block for device UI. It also returns a bounded curated StudyPack delta when the device
reports missing or stale pack revisions in `installed_study_packs`.

The current outgoing pack set is checked-in content from:

```text
agents/study-twin/examples/packs/
```

The server sends only the fixed files needed by the M5Paper runtime:

```text
manifest.json
concepts.jsonl
episodes.jsonl
rubrics.jsonl
```

It does not expose arbitrary filesystem reads or writes.

## Remote Access Model

The daily flow is LAN-first: M5Paper syncs to the paired host on the same network.

For office access, keep the home host as the sync endpoint and SSH into that host to inspect or
operate imported files. An SSH tunnel from an office laptop helps the human reach the home daemon,
but it does not make the office laptop reachable from M5Paper. If the actual sync app moves to
another machine, pair that machine as a new host or migrate the host state intentionally.
