# MIM Work Development Modeling Pack Policy

Status: active generation policy, 2026-05-28.

This note records how MIM work-development material on modeling and systems thinking is projected
into Biscuit StudyPacks after user feedback that shallow packs, obvious answers, and generic final
prompts are not useful on M5Paper.

## Feedback Incorporated

- M5Paper is a propagation and calibration surface for 10-15 minute sessions, not a long writing
  workstation.
- Each episode should carry enough method content to feel like a compact guide, not one slogan
  stretched over taps.
- Choices should be plausible working moves. The correct move should not always be first, and wrong
  moves should be tempting for understandable reasons.
- Learner-facing text should stay clean: no noisy prefixes mixed into content.
- Final episodes route follow-up branches rather than asking the same reflection everywhere.
- Cases should train working discrimination: object vs description, process vs system, target
  system vs creator, role vs actor, concern vs preference, level vs level.

## Projection

The first work-development modeling branch is five packs:

`mim-workdev-modeling-objects`
: Model, modeled object, description, property, behavior, creator, use situation, boundary, and
  working ontology.

`mim-workdev-models-communication`
: Epistemic state, attention object, confidence/source, communication as model change, role,
  concern, rhetoric, leadership, and silence as signal.

`mim-workdev-explanations`
: Explanation question, competing hypotheses, causal model, pre-experimental argument,
  observation, prediction, assumption, explanation quality, and explanation transfer.

`mim-workdev-object-dynamics`
: State, state change, temporal part, checklist, version, overlapping states, role shift,
  delegation, and attention retention.

`mim-workdev-systems-thinking`
: System of interest, supersystem, function in environment, system level, emergence, the systems
  mantra, multiple descriptions, concern/preference, and common system-identification errors.

## Composition Rule

Each pack should contain 12 episodes:

- 3-4 compact propagation episodes that introduce distinctions and a usable checklist;
- 7-8 diagnostic or contrast episodes with realistic work situations;
- 1 final routing episode whose choice says what the next branch should deepen.

The generated pack is a runtime projection. Durable source remains in the local MIM course corpus,
and each manifest must preserve source provenance.

## Acceptance

A work-development modeling pack is useful only if after the session the learner can do at least
one of these in a real project:

- name the object being modeled and separate it from descriptions, properties, actions, and
  creators;
- explain why a communication failed in terms of role, concern, object of attention, confidence, or
  source;
- create a better explanation by adding competing hypotheses and a discriminating observation;
- describe work as object state changes rather than as activity performed;
- identify a target system, nearest supersystem, system level, and a concern/preference conflict.

Completion alone is not evidence of usefulness.
